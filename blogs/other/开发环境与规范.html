    开发环境与规范

    Xiao_Ming 5/18/2021 其他知识

    今天commit代码的时候被师兄骂了,还好没 push,commit 内容写的像屎一样被喷了。所有的 commit 写的都是暂存 2 个字,毫无描述可言,而且后面 push 上去的代码各种格式问题,与团队不一致导致出现了很多的代码冲突。记录一下这个事情今后要小心注意。毕竟不是我一个人在爽,而是一个团队的协作。

    此外还有对一些生产工具的熟悉。发现自己之前一直在实验室用的 webstorm而去了很多家实习大伙们用的都是vscode，自己也需要遵循团队的规范并跟上时代的潮流。

    # Git 配置

    平时都会有自己的 Github✋，在融入团队后通常情况下团队又都会有自己的 GitLab✋

    这里要注意的是:

    • 给指定项目绑定 Gitlab 的用户和邮箱，一般为域账号和公司邮箱。注意当你需要提交 GitHub 项目时，请设置 GitHub 账户对应的的 username 和 email。

      git config --local user.name "your_name"
git config --local user.email "your_email@example.com"
git config user.name "your_name"
git config user.email "your_email@example.com"

    # git commit 规范

    commit 的时候应遵循一定的规范从而确保 commit 信息的有效性，以及在多人协作时候的规范性，包括易于项目的复盘以及 bug 的追踪和定位。

    语法: ${type}:${commit描述}

    # type

    用于说明 commit 的类别，通常为下列的 7 个标识

    • feat:新功能(feature)
    • fix: 修补 bug
    • docs: 文档(documentation)
    • style：格式(不影响代码运行的变动)
    • refactor：重构(既不是新增功能，也不是修改 bug 的代码变动)
    • test: 增加测试
    • chore：构建过程或辅助工具的变动
